大多数开发者最开始使用 coding agent，都是为了写代码：查看代码仓库、生成修改差异、运行测试，以及提交 pull request。

这依然是 Codex 的核心能力。但电脑上的很多工作，本来就已经通过代码间接完成了：执行 shell 命令、浏览网页、调用 API、导出文档、响应事件、触发自动化流程。随着这些能力逐渐开放给 Codex，它就不再只是狭义上的编程助手，而更像是一个帮助你完成电脑工作的系统。

  • 使用侧边栏，让用户审阅代码、文档、演示文稿和其他产物

  • 文档审阅线程

  • 文档审阅线程

  • 专门用于外部监控的线程

这些线程是持久工作区，不是短对话。Codex 可以在之后继续回到这些线程，保留之前的决定、偏好和工作上下文。否则，这些内容每次都需要重新建立。

置顶线程快捷键让这种方式更实用。Command-1 到 Command-9 可以直接跳转到已保存的线程。

## 语音输入

语音输入的价值在于，它能记录一个想法还没有被整理成正式文字前的粗略版本。

Codex 内置了语音输入。它特别适合那些说出来很自然、但打字时很难整理清楚的模糊起点：

我记得 Slack 里好像有个叫 Ben 的人提过这个。

我不记得细节了。

请去查一下。

对于一个能够搜索、收集上下文并汇报结果的代理来说，这通常已经足够。

它也很适合在任务尚未完全成形之前，进行两三分钟的想法倾倒。

会议转录也可以用同样的方式处理。一份原始会议记录或口述规划笔记，往往比简短摘要更适合作为材料，因为它保留了不确定性、重点和未完成的想法。

## 即时调整和任务排队

语音输入如果配合对当前任务的明确控制，会更有用。

即时调整： 在 Codex 正在执行任务时，于当前步骤完成之前打断它，并给出新的方向。

当代理正朝着不合适的方向推进，需要在它完成之前修正时，这很有用。比如在审阅网站时，用户可以一边在侧边栏里标注界面，一边打断当前工作：

  • 让这个小一点

  • 这两个元素之间的间距感觉不太对

  • 这段文案不对

任务排队： 给 Codex 添加当前步骤完成之后要处理的工作。

任务排队和即时调整不同。它不会打断正在进行的任务，而是把下一个任务加入队列。用户可能会说：

等工作完成后，把预览链接发给 Slack 里的审阅人。

即时调整改变 Codex 现在正在做什么。任务排队改变 Codex 接下来应该做什么。两者都能让用户在工作推进过程中保持参与。

## 工具和可操作范围

一旦线程具备连续性，接下来的问题就是它能操作什么。Codex 可以逐层向外扩展：

  • $browser：用于侧边栏里的内置 browser，Codex 可以检查并标注网页界面

  • @chrome：用于依赖用户 Chrome 登录状态的浏览器工作流

  • @computer：用于只能通过桌面图形界面完成的工作

$browser 适合侧边栏里的网页审阅。@chrome 适合依赖用户 Chrome 上下文的登录态浏览器工作。@computer 适合那些只能通过桌面图形界面完成的任务。

MCP servers 和连接器会把同样的思路扩展到工作流的其他部分。Slack、Gmail 和 Calendar 很重要，因为很多重要任务最开始并不是代码，而是消息、邮件或日程问题。

技能让重复工作流可以复用。当一个工作流被证明有用之后，可以把它打包成一个技能。这样 Codex 下次就能再次执行，而不需要重新学习整套流程。

## 随时继续工作

Codex mobile app 改变了用户必须坐在桌前的时机。一个任务可以在 Mac 上开始，因为文件、权限和本地环境都在那里；之后用户可以通过手机继续查看和参与。

这在很多小场景中很有意义。用户可以在 Codex 执行较长任务时离开桌面，在外面回答问题、批准下一步，或者在回到电脑前重新指引线程。原本的本地环境仍然保留，用户不必一直守在电脑前。

## 自动化

自动化会按计划运行 Codex 工作。当某个重复任务应该从某个工作区重新开始时，可以使用计划自动化，例如日报或定期代码仓库检查。当计划任务应该回到一个已有对话，并继承其中的运行上下文时，可以使用线程自动化。

线程自动化： 类似心跳一样的周期性唤醒机制，会按计划回到同一个 Codex 线程。

置顶线程很有用，但它们仍然需要等待用户回来。线程自动化可以每几分钟或每几小时检查一次某件事，持续运行直到满足某个条件，并随着时间调整频率。

一个 Chief of Staff 线程可能每 30 分钟运行一次：

每 30 分钟检查 Slack 和 Gmail，看看有没有需要我关注但尚未回复的消息。

帮我判断哪些事情最重要。

如果有人问我问题，请尽可能深入研究答案，并为我起草回复，但不要发送。

当用户回来时，收集上下文这部分耗时工作通常已经完成。最终发送什么，仍然由人决定。

线程自动化也适合反馈循环。它可以关注 pull request 评论、Google Docs 评论或 Slack 回复，并在用户离开时继续推进相关工作。

设想一个动画工作流：审阅人在 Slack 里分享了一段视频。线程自动化可以按计划检查这个线程；当评论出现时，渲染更新版本，并在同一个线程中回复和提及审阅人。如果某个集成无法完成最终上传，桌面自动化可以通过图形界面完成这一步。

这个流程横跨 Slack 反馈、代码库渲染和桌面自动化上传。

## 目标

当任务有真实完成标准，并且代理可以持续朝这个标准推进时，目标最有价值。一个较弱的目标是：

目标： 有明确完成线、Codex 可以随时间持续推进的长期任务。

按这个 Markdown 文件里的计划实现。

更强的目标应该有可衡量的成功标准。

例如，工程师可能要把一个内部工具从 Python 迁移到 Rust。他可以建立新的目录，定义目标，并明确完成标准：新的实现只有在单元测试通过后才算完成。

一个目标会把持续执行和验证方式结合起来。用户定义期望结果、停止条件，以及用来判断 Codex 是否更接近目标的信号。

有用的验证方式包括：

  • 测试套件

  • 性能基准

  • bug 复现方式

  • 验证矩阵

  • 必须持续通过的端到端工作流

有野心很重要，但没有验证方式，它就只是愿望。

## 侧边栏

侧边栏会把工作保留在产生它的对话旁边。用户不需要导出产物再切换到其他环境，而是可以在原处审阅。输出可能是代码，也可能是演示文稿、PDF、浏览器页面、表格，或过程中创建的其他产物。

它特别适合四类工作：

  • 查看产物

  • 标注需要修改的内容

  • 操作网页界面

  • 审阅改动

侧边栏让用户可以在原处审阅 Markdown、电子表格、数据表、文档和幻灯片。他们可以检查、标注和修改产物，而不需要离开当前工作流程。

## 共享记忆

当长期线程能够在单个对话之外共享记忆时，它们会更有用。

共享记忆： 存储在单个线程之外的持久上下文，让未来的工作可以从明确且可审阅的内容继续。

一种持久做法是把长期线程连接到一个 Obsidian vault。实际使用时，这意味着一个由纯文本文件组成的文件夹；这些文件长期来看容易检查、编辑、移动和保存。团队可以把这个文件夹存放在云存储、Git、Dropbox、Google Drive 或其他适合自己工作流的同步层中。

一个 vault 可能长这样：

vault/

├── TODO.md

├── people/

├── projects/

├── agent/

└── notes/

在顶层，AGENTS.md 可以定义 Codex 在了解更多人、项目、决定和待处理事项时，应该如何更新这个工作区。

不要照搬某一个 vault 结构。应该告诉代理：持久上下文应该放在哪里，什么上下文值得保留，以及什么时候不应该制造无意义的变动。

一个实用的 AGENTS.md 可能会写：

- Treat ~/vault as durable work memory.

- Prefer canonical notes over note sprawl.

- Route TODOs, people, projects, daily summaries, and scratch notes explicitly.

- Preserve decisions, blockers, owners, dates, and useful links.

- If nothing meaningful changed, do not churn the vault.

代码仓库保存代码。Vault 保存持续变化的上下文：涉及哪些人、发生了什么变化、什么受阻、哪些事情需要跟进，以及那些原本会在会话之间消失的信息。

重要上下文不应该只存在于对话记录里。应该把它写到某个地方，让下一个线程能继续使用。

Codex 也有官方记忆功能，位于 Settings > Personalization > Memories。它们可以记录偏好、重复工作流和常见问题，作为一种本地回忆层。它们是对显式书面上下文的补充，不是替代。Chronicle 也朝着同一方向发展，帮助 Codex 从近期屏幕上下文中建立记忆。

## 从代码向外延伸

Codex 仍然从代码开始。但围绕代码的更多工作现在都可以通过同一个系统触达：MCP servers、browser 界面、桌面控制、线程自动化和可审阅产物。

这改变了控制方式。即时调整会改变正在进行的工作。任务排队会安排接下来要做的事。线程自动化会在用户离开时保持线程活跃。目标会加入明确的完成线，让 Codex 能持续朝目标推进。

Codex 现在可以把一个工作流从指令带到执行，再带到产物审阅，即使这项工作已经离开了代码仓库。

在征服中国的过程中，满洲民族迅速从落后的狩猎部族文明阶段进化到农耕帝国的文明水平。由于没有受到腐朽的理学思想污染，且内部特殊利益集团尚未发育成熟，这个处于上升期的民族表现出旺盛的活力，诸多优秀的帝王将相不断出现，并且善于任用汉民族优秀人才和政治制度来治理中国，这让他们在短时间内掌握了大约相当于中国秦汉时期的国家治理能力——在某些方面超过了秦汉，比如有了科举制度；在某些方面则要落后于秦汉，比如对思想文化的控制远远不如汉朝开明自由，但总体上是差不多。秦汉时期，也是汉民族积极进取，建立帝国体制和强大的军事力量，快速开疆拓土的时代。

但对汉民族而言，这却是一个巨大的退步和长达两百多年的噩梦。中华文明因此倒退了至少一千年，某些方面甚至倒退了两千年——比如思想文化方面的自由程度连春秋战国时期的水平都达不到。

对清朝的疆域广大、统治时间长和它对人民的屠杀、思想文化的控制等如何做一个综合的评价和衡量——也就是孰轻孰重？是一个很复杂的问题。

我们继续从文明的视角来思考这个问题。

就目前为止的人类文明发展史来说，我们有几个基本的判断。比如，从游牧和渔猎文明走向农耕文明是一种巨大的进步，这一条不赞成的人应该不多。第二条，从农业文明走向工业文明，也是一个巨大的进步。这一条应该也没有什么问题。

中华文明在从游牧和渔猎文明走向农耕文明的过程中，是领先全世界的。不仅进入农耕社会的时间领先，更重要的是在农耕时代建立了古代社会最发达的政治体制、最辉煌的经济成就、最长久的和平稳定，并供养了数量最庞大的人口，开拓了广阔的疆域。这是值得每一个中国人骄傲的。

不过，在从农业文明走向工业文明的过程中，中华文明大大的落后了。这种落后，是什么原因造成的？

工业革命为什么没有发生在古代经济最繁荣的中国？也就是好多人喜欢讨论的所谓“李约瑟难题”——英国学者李约瑟（Joseph Needham，1900-1995）在其编著的15卷《中国科学技术史》中正式提出此问题，其主题是：“尽管中国古代对人类科技发展做出了很多重要贡献，但为什么科学和工业革命没有在近代的中国发生？”1976年，美国经济学家肯尼思·博尔丁称之为李约瑟难题。

这个所谓的难题其实一点也不难。之所以会成为难题，是因为它的答案，很多人出于利益原因不愿意讲，甚至还想办法不让人讲。

只要讲清楚明清换代之际，中国人所遭受的人类文明史上空间绝后的大屠杀 ，这个难题其实就不是难题了。

明清换代，人口从四个亿给杀到还剩不到一个亿，尤其是在经济文明最发达的江南华南地区出现大规模屠城，城市精英人口损失殆尽，面对野蛮暴政稍有点反抗精神的人都会被全家老小一起杀掉，沿海五十里内的人民和城镇被尽数毁灭，这种情况下，哪里还有可能爆发什么工业革命呢？如果英国在十八世纪刚刚发明珍妮纺纱机的时候，突然遭到北欧海盗的入侵并被征服，在各大城镇疯狂屠城，令其损失一半以上的人口，它能爆发工业革命吗？

中国未能爆发工业革命的直接原因，如果只挑选一条，那就是明清换代。一个处在工业革命爆发前夜的文明，突然遭到北方蛮族的入侵，对其人口进行了毁灭性的屠杀。《天工开物》等诸多本土科技典籍因此失传，《几何原本》等西方科学名著的翻译被迫中断。被屠杀的人口中，也许就有类似于牛顿、笛卡尔、莱布尼茨这样的人物，或者是可以启迪他们发明创新的师友，还有很多可以发明诸如珍妮纺纱机、蒸汽机的能工巧匠。总之，就是整个东南沿海的手工业创新体系被彻底摧毁。工业革命在短期内（一两百年）爆发的可能性自然就消失了。

当然，这只是直接原因，它是背后还有很多更深刻的间接原因。核心还是汉民族内部精英的集体堕落，不管是理学思想的僵化、官员的腐败、财富阶层的自私、东林党的无耻、孙承宗和袁崇焕的愚蠢、农民起义军的不团结和战略失误……这些都值得深刻反思。也包括西方殖民活动带来的白银货币冲击、中国的背靠大陆面向大海的地理特征等客观因素。但最后所有原因汇总起来，让北方蛮族入侵成功，杀遍中国，这是一个总阀门——它最终切断了、关闭了中华文明自主进入工业文明的发展路径。

我们分析历史无论如何强调各种间接原因、客观因素，这个总阀门绝不能跳过去不讲或者有意无意的淡化。只有把这个总阀门的前因后果讲清楚，中国历史的脉络才能说清楚、讲透彻。也许会有人说我们记仇，或者煽动历史仇恨之类的。那毕竟是古代社会的事情，也过去好几百年了，记仇是不应该的。关键在于：不讲明白，就会不停的有人乱甩黑锅，把中国近代落后的根源甩到中华帝国的皇帝专制制度、科举考试制度、朱元璋杀功臣、明朝海禁、中国人的劣根性之类的东西头上。我们可以不记仇，但不能乱背锅。这个锅背错了，我们反思历史的方向就会发生错误，会对我们未来道路的选择产生深远的影响。
