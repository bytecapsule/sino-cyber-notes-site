Michael Pettis：中国“软预算约束”夸大了GDP

  1. 外国对于中国“总是能实现”5%之类的目标GDP增长率感到奇怪。Pettis在北大光华任教多年，对中国经济运行比较了解，解释说这是因为中国有保证目标实现的手段，所以中国GDP增长率不是一个预测，而是一个目标。但Pettis认为，这个手段是“进行非生产性投资而无需正视成本”，不是好事。

  2. 具体来说，当中国发现需要补上GDP目标，就可以进行投资，最后GDP增长率就达标了。其它国家没法这样做，因为预算是早就定下的，是“硬预算约束”，而且全球GDP核算都是基于“硬约束”体系来设计的。而中国是“软预算约束”（ Soft Budget Constraint，科尔奈发明的术语），许多企业没有预算也可以找银行借钱，把投资做成。另外一个问题是，建成的资产，价值立刻计入GDP，不需要靠销售产品来实现，因为投资本来就需要时间。但等到实际上基建荒废、工业园区没有企业入驻、产品销售不好，就会发现原本的投资是失败的。这意味着当初计算的GDP是无效的，夸大了GDP。

  3. 理论上来说，“硬预算约束”是资本主义的纪律机制，防止经济主体持续滥用资源。经济主体必须严格控制亏损额度，只能花自己挣的钱，或者以可信的方式借贷；如果亏损又无法融资，就破产了。中国不一样，很大一部分（且占比不断上升）经济领域都处于软预算约束，经济主体无论其资产状况和盈利预期如何，都能持续获得信贷支持。银行等贷款人假定，借款的出事了也将获得救助。中国资源被滥用的唯一限制是地方与中央政府本身的信用状况。

  4. 因此理论上来说，中国在持续将投资错误地配置到非生产性项目中，且相关损失未得到正式确认，这种软预算经济体的GDP，无法与硬预算经济体的GDP数据比较。后者会确认损失并从GDP中扣除，而前者可能根本不确认也不扣除。

  5. Pettis近年来一直对中国的投资想不通，多种角度说“有问题”。这回又换了个“软预算约束”的说法，还是差不多的感觉，听上去有道理，但一琢磨就还是老一套的“不能接受”。这个“软预算约束”，恰恰是中国投资能力强劲，能快速发展的最大绝招之一。什么方法都有利有弊端，软预算约束是有政府冲动投资、投资失败的大毛病，全国不少地方建的工业园区不成功。那有没有很成功的例子？成功案例应该是全球最多，占据了全球50%的制造业产出，有海量案例。

  6. 至于GDP核算，不成功的投资也计入GDP了，确实有这个问题。如果有一个“GDP核算理想国”，那中国肯定还有差距。但GDP质量，并不只是中国有问题，全球GDP质量差得要死的国家多得是。资本主义体系的GDP质量有一个本质问题，对于什么是“价值”说不清。如中国建了一条铁路，发达国家投资10倍建了长度和速度一样的铁路，后者在GDP核算中是10倍，如何解释？用PPP法也只能缩小2倍之类的差距。

  7. 另外，中国存在大量投资相关的GDP低估问题。如提前多年用10亿低成本搞了一个基建，非常成功收益极好，如果现在去投，需要投100亿才做得成。这种提前投资，在“软预算约束”中容易实现。而且也没有办法去给这种成功的提前投资估值，当年的GDP只会是10亿。一个荒谬的办法可以直接增加GDP：先用布把这个10亿的基建盖起来，说没有了，然后左手倒右手宣称投资100亿，再把布拿开就实现了100亿GDP。因此，Pettis等人对于中国投资的攻击，是发现了问题，但还没有想通。GDP等经济学概念体系，存在本质的问题。

其次是Cowork+Opus 4.6，再其次是Claude Code+Opus 4.6，再其次是Claude Code+k2.5，其他的我基本就用的很少了，大家可以自己去试试。

需要注意的是，Claude Opus 4.6产出的内容，很多都有幻觉，但同时幻觉又是创意的前提，所以我一般不会在Skill里面强行的抑制幻觉，而是把AI生成的内容，扔给Codex+GPT-5.4，来检测是否含有事实不符的地方。

如果你不用Codex，用ChatGPT也可以，记得开启GPT-5.4的Thinking模式，GPT-5.4的幻觉率，真的非常的低，低的可怕。

我觉得精准性上远超其他的所有的模型。

用法大概就是这样，但是我非常的不推荐你拿到手就直接用，因为那只会让你模仿我，但是永远不会超越我。

所以，这篇文章的后半部分，我还想再聊两个我觉得是更核心的内容。

如果说，我的卡兹克风格创作Skill是鱼的话，那后面这两个话题，我觉得就是渔。

希望对大家真的有用，我也非常推荐大家看完，可能会有点长，但，一个一个来说吧。

一. 如何构建和迭代你自己的创作Skill

首先我想说明的一个事，就是其实很多人对Skill有误解。

Skill，或者说一个做创作的Skill，从来都不是一次成型的。

你自己有新知识、有新的成长，Skill理论上需要跟着你一起迭代这种东西我就不说了。

而是就单纯的构建一个创作Skill，我觉得也从来不应该一次成型的。

你可以回想一下你看过的市面上那些XX内容生成Skill，想一想很多都是怎么做的。

其实大部分人的做法是，把你发过的内容、视频脚本、文章啥的丢给AI，让它总结你的风格特征，然后蒸馏成所谓的一个Skill。

但它有一个致命的问题。

就是AI总结出来的，是它认为的你的风格，而并不是你实际的风格。这两个东西之间的gap，其实远比你想象的要大得多的多。

所以我自己迭代Skill的做法，其实会稍微有一点点不一样。

第一步，我还是会跟之前一样，扔两三篇我自己觉得比较有代表性的文章进去，还有我自己的曾经给内部写的方法论白皮书，然后总结成一个所谓的卡兹克创作Skill，这个第一步，其实完全没有变化。

第二步，让AI按照当前版本的Skill，根据你详细的要求、Prompt、还有给过去的素材，直接帮你写一篇文章，这一篇说真的，几乎不可能发出去，因为它一定有很多地方，读起来你会觉得，嗯，好像哪哪都差那么一点意思。

第三步，自己动手改，你可以把AI生成的这篇稿子当做参考，在这之上，直接重写一篇都行，只要保证主题是一致的，你想表达的观点是一致的。你一定一定要动手改。

第四步，把AI生成的版本和我重写的版本，同时丢回给AI，让它自己分析这两个版本之间的差异。哪些地方它写得太正经了，哪些地方它用了我绝对不会用的词，哪些地方它的节奏太平了，再把这些差异迭代回Skill里。

然后，重复以上第二步到第四步的步骤，重复3～4轮。

你这个创作的Skill，终于趋近于稳定了，它终于知道你想要的风格是什么样的了，它终于逐渐的，磨平了你们中间的GAP了。

然后这块我有个建议，就是这个轮数从来不是越多越好，我自己经验和体感下来，其实建议3～4轮左右就差不多了，不需要搞个10～20轮，那样会容易过拟合，反而限制AI在一些思维上的发散和创新。

而且你要不要指望AI能帮你直接写出一篇100%可以用的内容，永远不要相信，他能生成一个70～80%我能直接可用的，我都觉得今天是上天善心大发了。

因为就连你自己，在不同的时间点，不同的情绪状态下，写出来的东西都是不一样的，都是有那个情绪表达在的，你没办法要求AI比你自己还稳定，那样，活人感反而没有了。

这个就是如何构建和迭代你自己的内容创作Skill的核心。

然后我想强调的Skill里面另一个非常重要的东西，叫做自检系统。

前两天Claude Code源代码泄露的事情大家肯定都知道了，当时很多人都在聊，哇Anthropic自己也用AI写代码写的真牛逼，里面有那么多可以学习的，哇Claude Code有这么新功能。

但是我跟朋友聊的过程中，我们其实比较惊讶的点在于，Anthropic是如何能做到AI产出的代码质量这么稳定的，我们其实一致觉得，Anthropic那套代码的质量评估体系还有测试系统，才是真正的最牛逼的。

这个牛逼的程度，是OpenAI现在都还完全没有赶得上的，从产品的迭代速度就能看出来了。

这个思路，其实跟我们做内容创作的Skill是有异曲同工之妙的。

所以我在我的Skill里面，也搞了一套四层自检体系。

第一层是硬性规则扫描。像代码的语法检查一样，检查有没有用禁用词、有没有用禁用标点、有没有出现结构性套话，这些是最基础的，出现了就一定有问题。

第二层是风格一致性检查。开头是不是从具体场景切入的，节奏有没有长短句交替，口语化表达够不够，有没有那种一句话独立成段的断裂效果。

第三层是内容质量检查。每个观点有没有具体的场景支撑，知识是不是聊天感的方式出来的，有没有从具体事件连接到更大的文化参照物。

第四层是活人感终审。这一层最主观，也最重要。就一个核心问题，读完这篇文章，你觉得这是一个有见识的普通人在认真跟你聊一件打动他的事，还是一个AI在给你输出信息？

如果任何一段读起来感觉AI味重了，就需要返工。

大概就是这样。

构建+自检+迭代，就是我觉得一个正确的Skill的构建流程。

你当然可以把我的卡兹克创作Skill拿去用，但是我觉得这样其实能帮你完成一些事情，但并不能百尺竿头更进一步。

我非常推荐大家基于这个Skill的底子，去构建你们自己内容创作Skill。

一千个人用这套方法，能做出一千个不同的Skill，服务于一千种不同的内容风格。

这才是我最想看到的盛况。

二. AI到底应该如何正确的辅助你的创作

接下来我想聊一个更敏感的话题。

就是，我到底怎么用AI辅助我做内容创作的。

去年，我在代表《流浪地球3》的AI团队接受外部媒体采访的时候，我曾经表达过一个组内的观点和共识，就是在球3的时候，我们有一个原则就是，AI内容不进成片，但是在成片之前，有大量的工业化的体系，是我们在用AI去进行辅助加速生产的。

但是，那毕竟是《流浪地球3》，是中国电影工业体系的巅峰，是科幻电影的桂冠，AI，还远远达不到剧组的要求。

但是回到我自己身上，我就是一个写文章的，AI写的东西，一丁点都达不到人类的要求？

鬼特么才信。

坦率的讲，都2026年了，一个公众号上写AI内容的AI博主说自己完全不用AI辅助内容创作，你信吗？

但我还是想把这个事情说清楚。

首先我想表达的是，AI辅助创作，是在你的创作流程中，有很多环节和类型，是AI可以加速进去提效的，从来不是说，你扔一个话题进去，AI就能直出帮你写完全文，这个想都不要想，除非你对内容没有一点要求，或者完全不在乎你的读者和观众的阅读体验，那你可以这么干，但是我自己只能说，极度不推荐，这会让你自己在内容创作能力上变废掉。

用AI辅助创作，和直接让AI替你写完内容，是完完全全不同的两件事，这个是理解我下面所有内容的前提。

这块我也说一下之前我文章错别字的问题，很多人都在说我是AI生成刻意再改错几个字冒充人类，我只能说我有病我才会这么做，我这辈子最接受不了的就是造假和欺骗，AI生成改错别字冒充人类欺骗读者，这种行为在我这就是绝对的触碰底线了，是我绝对不可能接受的。

所以其实之前的错别字，都是因为我比较懒，写完的东西我真很不想检查一遍只想赶紧发了，所以才会漏了一些错别字在里面。

但是我也深刻反思了，我觉得这样不好，所以我现在都会用Agent帮我过一遍错别字审查。。。

所以最近一个月，大家应该会看到，我的错别字几乎销声匿迹了。。。

话说回来，说回到AI辅助创作这块来。

我先说什么东西是我绝对没办法用AI来做的。

那就是，实测内容和教程内容。

这个大家肯定都能理解，你要写一篇新模型或者新工具或者新功能的体验和评测，你得自己去用，自己去体验，自己去踩坑。

甚至我们经常是首发，连网上的参考都找不到一点，只能完全自己来，所以这种内容，是AI完全做不了的，AI不能替代你亲自体验，也不能替代最终判断。

这些内容，AI基本辅助不了一丁点。

但是内容创作中，有几个流程，是我经常会使用AI辅助我来进行加速的。

分别是找证据写观点、写类比和比喻、确定了角度和框架之后的扩写。

这也是我现在用的最多的用法。

先说第一个，找证据写观点。

比如说，根据某个事件，我写了一个观点，我觉得这个观点是对的，但我想找到一些过去的历史事件、学术研究、或者文化现象来佐证它。

在以前，我可能需要翻大量的资料，或者纯靠自己的阅读积累来硬顶。

但现在，我会让AI帮我找。

我的做法一般是这样，让AI给我遍历能证明我这个观点的五个案例，以及能反驳我这个观点的五个案例。

然后我自己来判断，挑出最合适的，揉进我的文章里。

你说这个过程用了AI辅助吗？当然用了。

但最后文章里的观点是谁的？是我的。

AI帮我找的是弹药，但开枪的方向是我选的。

千万不要丢了自己判断的能力。

第二个，找类比和比喻。

这个比较好理解，有时候我脑子里已经有一个特别好的比喻了，那我就直接把我写好的部分扔给AI，让它按这个比喻的方向来续写。

但有的时候确实没灵感，那我就让AI根据我想表达的意思，帮我生成几个比喻或者类比的候选。

有的候选特别烂，一眼就能看出来是AI的那种比喻。

有的还不错，能给我一些启发。

最终用哪个，怎么改，还是我会自己来。

然后最后，就是确定了角度和框架之后的扩写。

这个我想举一个非常真实的例子，是我自己之前写的，一个我自己非常非常喜欢的文章：《淘宝上卖9块9的DeepSeek，让我看到了被折叠的魔幻世界。》

我到现在还记得，当时我写到凌晨1点的时候，前半部分已经写完了。

前半部分是纯粹的我们自己的经历，下单买DeepSeek、收到了几个免费网址的链接、一脸懵逼的去看评论区、看到那句“不用买会员了也”的时候鼻子一酸。

这些东西你知道的，几乎不可能用AI的，因为全部是我自己的体验，我自己在那个瞬间真实的情绪反应。

但到了后半部分，我卡文了，不知道该怎么写了。

我想表达的核心主旨是有的，就是想表达一下信息差，但是实在不知道用什么角度切入，我真的没有想好。

当时又很晚了，于是我让AI帮我写了一些后半段的角度切入。

但没有一版是我满意的。

真的，大家可以自己去试试，你一旦让AI聊信息差这种话题，它就变得过拟合到极点。出来的那些角度，我闭着眼睛都能猜到它会怎么切入，什么割韭菜是不对的，什么我们应该技术普惠，真的很没有意思。

太普通了，太正确了，正确到没有任何信息增量。

说真的，后来凌晨2点半了，我真的想不出来，甚至已经准备放弃，第二天不发了。

然后我抬头看了一眼书架。

看到了那本《北京折叠》。

北京折叠，被折叠成了三层。

在这个时代，信息差，让我们的空间，让我们的信息，也被折叠成了三层。

那一秒钟，我感觉角度有了。

第一空间，信息的创造者和定义者。第二空间，信息的追随者和消费者。第三空间，信息的隔绝者。

我把观点和三层空间写好之后，我丢给了AI，让它按我定的角度去扩写每一层的内容。

它扩写得非常快，质量也挺好。因为方向、框架、角度全部已经确定了，AI只需要在这个骨架上填肉就行了。

后面我就直接改吧改吧用上了，大概3点半，那篇稿子最终定稿了。

这个凌晨2点半的故事，就是我想说的一切。

AI能帮你做的事情，是在你定好方向之后，极大地加速你的产出。

AI做不到的事情，是那个凌晨2点半，你抬头看书架的那一秒钟。

那一秒钟，是你在一个特定的空间里，一个特定的疲惫状态下，目光偶然扫过一本书的书脊，然后大脑里突然产生了一个跨领域的概念嫁接。从一个淘宝9.9元的商品，跳到了一本科幻小说里的社会分层。

这个东西，我至今没有任何Prompt能复现。

我把它称为，独属于人类的。

神之一手。

写在最后

所以回到那个很多人可能会关心的问题。

既然你有一个写作Skill，那你以前的文章是不是都是AI写的？

我觉得，如果你认真看完了我上面讲的这些，你自己心里应该已经有答案了。

但我还是想把话说得更直接一点。

“这篇文章是不是人写的”，在2026年的今天，我真心觉得它已经是一个过时的问题了。

今天任何一个创作者，如果他说自己100%纯人工创作，不用任何AI辅助，要么他在说谎，要么他在跟时代较劲。

查资料你会用AI，找灵感你会用AI，校对你会用AI，甚至你做封面图都可能用AI，我们的创作流程中，有大量的AI辅助在里面。

AI已经像空气一样渗透到了创作的每一个环节里了。

这个时代，真正值得问的问题，并不是“是不是人写的”。

而是，这个内容背后，到底有没有一个真实的人在做判断，有没有一个人，在做真实的情感表达。

有没有一个人，在凌晨2点半实在想不出角度的时候，没有凑合着用AI那个平庸的版本，而是选择抬头看了一眼书架。

有没有一个人，在AI给出五个候选类比的时候，说不对，这些都太普通了，然后自己去想。

有没有一个人，在看到淘宝评论区那句“不用买会员了也”的时候，是真的鼻子一酸，说我一定要为他们发声，而不是在表演鼻子一酸。

这些判断、这些选择、这些真实的情绪反应，是任何AI、任何Skill都没办法生成的。

也是我觉得，我把这个Skill开源之后，任何人拿着它都复刻不了的。

我依然还是我。

这是属于卡兹克的印记，这也是我的灵魂。

我开源了这个Skill，我也不希望大家去模仿我，而是可以在这之上，去构建一个新的“你”。

最后。

给自己打个广告，我自己办的万人的AIFUT大会，在明天，终于就要开幕了。

更关键的是，这些技能在使用过程中会自我迭代。如果 agent 在执行技能时发现了更好的方法，它会自动更新技能文档。有 Reddit 用户反馈，agent 在两小时内自动生成了三份技能文档后，重复性研究任务的速度提升了 40%。

OpenClaw 也有技能系统，但主要依赖人工编写和社区贡献的技能市场 ClawHub。Hermes 这边等于把“写技能”这件事也交给了 agent 自己。

【3】记忆体系的差异

两者都声称有跨会话记忆能力，但实现方式不同。

Hermes 用 SQLite 配合 FTS5 全文检索，把所有历史对话存下来，需要时通过搜索加 LLM 摘要召回。它把记忆分成两层：一层是常驻的关键信息（写在 MEMORY.md 里，每次对话都注入上下文），另一层是全量历史检索（容量无限，按需调用）。

OpenClaw 同样用 Markdown 文件存储记忆，也支持 SQLite、QMD 等多种检索后端，通过 memory_search 工具做语义或混合检索。它的特色是在上下文压缩前会执行一次静默记忆写入，防止压缩丢信息。官方的设计原则是"无隐藏状态"，记忆就是工作区里可读可编辑的文件。

两者的核心差异在于：Hermes 更强调对全量历史对话的检索和总结能力，像给 agent 装了一个搜索引擎；OpenClaw 更强调记忆的透明性和可编辑性，用户能直接看到和修改 agent 记住了什么。

【4】安全思路也不一样

Hermes 搞了一套五层纵深防御：用户授权、危险命令审批、容器隔离、凭据过滤、上下文注入扫描。默认对高风险操作（比如执行终端命令、写文件）要人工审批，超时未批准就自动拒绝。

OpenClaw 这边则更强调信任模型和配置审计。它提供了 openclaw security audit 命令，可以一键扫描网关配置的安全隐患。但 OpenClaw 在安全方面的历史记录不太好看，今年 2 月被曝出多个高危漏洞，13.5 万个实例暴露在公网上，技能市场也有超过 300 个恶意技能被发现。

【5】要不要换或者选哪一个

如果你日常用的 Agent 已经顺手，没必要。如果你之前的龙虾主要是 claude code 的授权现在用不了可以试试这个，但不能保证继续用多久。如果喜欢折腾想试试不同的选择，也可以试试。

最近经常被问到这样一个问题：像 Zig 这样定位的语言多吗？到底该怎么选？

其实放眼整个系统级编程领域，虽然新语言层出不穷，但真正像 Zig 这样，试图完全对标“Better C”、坚持极简主义且手动管理内存的选手，其实并不多。大多数现代语言都走向了复杂的“重型武装”路线，而 Zig 却选择了一条返璞归真的小径。

我们要把这些语言分个类，第一梯队里最绕不开的自然是 Rust 和 C++。Rust 就像是一辆自动驾驶的装甲车，它把“安全”做到了极致，编译器会按着你的头让你遵守规则，虽然极其可靠，但那陡峭的学习曲线和繁杂的语法，往往让习惯了自由的 C 程序员感到窒息。而现代 C++ 则像个无所不能的变形金刚，功能强大但也极其臃肿，历史包袱沉重，构建系统更是无数开发者的噩梦。

在第二梯队的小众领域，也有几位有趣的挑战者。比如 Odin，它专为游戏开发设计，虽然也主张无 GC 和手动内存管理，但在通用性和元编程的统一性上，还没能达到 Zig 的高度。还有传说中的 Jai，那是《Braid》作者为了革 C++ 的命而造的屠龙刀，可惜至今未公开发布，只存在于传说中。至于 V 语言，虽然号称简单如 Go 性能如 C，但早期宣传的争议让它在严肃的系统编程圈子里还需要时间来建立信任。Nim 虽然语法优雅，但本质上还是依赖 GC 的，并不适合那些对延迟极度敏感的硬实时场景。

所以回过头来看，Zig 的位置非常特殊，甚至可以说有点孤独。它不像 Rust 那样试图用复杂的借用检查器来接管一切，也不像 C++ 那样无休止地堆砌特性。Zig 更像是一辆现代化的手动挡赛车：它保留了 C 语言那种“得心应手”的底层控制感，没有隐藏的内存分配，没有隐式的代码执行，但在仪表盘上加装了强大的编译期计算能力和现代化的类型检查。

对于那些受够了 C 语言的宏定义和 CMake，但又不想被 Rust 编译器反复折磨的底层开发者来说，Zig 或许是目前唯一且最佳的舒适区。它在“保持简单的控制力”和“提供现代化的工具链”之间，找到了一个微妙的平衡点。这大概就是为什么在众多挑战者中，Zig 能脱颖而出的原因吧。
