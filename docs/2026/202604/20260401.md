经产省通商白书2024 + JBIC第36回调查提出一句话，经产省海外事業活動基本調查：全球日系制造业现地法人约11,000社，其中约8,400社（~80%）在亚洲。但投资额只占40%。这些投资不是少数大工厂，而是8400家小据点组成的蛛网——系列化（keiretsu）的海外延伸。

通商白书进一步指出：海外调达中，关联公司（资本关系のある関係会社）的比例特别高，尤其在亚洲。这正是keiretsu在海外的延伸——不是自由市场采购，而是母公司带着一整套供应商一起出去。

这个投资模式和北美完全不一样，北美也是40%投资额，但同样是40%的投资额，亚洲靠8,400家小据点消化，北美靠800家大工厂消化。亚洲模式是"撒芝麻"，北美模式是"码砖块"。

在亚洲这边投资典型路径是这样的：丰田决定在泰国建厂，电装、爱信、丰田合成、丰田纺织跟着一起去。每家一级供应商又带着自己的二级、三级供应商。结果：丰田1家 → 电装/爱信/丰田合成等Tier1（6-8家） → 每家Tier1带3-5家Tier2 → 再往下...

所以，日本的海外投资不是一座座孤立的工厂，而是一张从日本本土延伸出去的关系网络的海外复制。

以丰田进入泰国为例，完整的蛛网织法：

第一波（1962-1970s）——母厂落地

丰田自己建Samrong工厂。这时候零部件基本从日本进口，本地调达率很低。

第二波（1980s）——Tier1随伴

丰田产能扩大后，核心供应商被"请"过来：电装、爱信、丰田合成、丰田纺织、捷太格特（JTEKT）。这些企业不是自己想去泰国，是丰田需要它们在当地供货。日语叫 「随伴進出」——跟着老大一起出征。

第三波（1990s-2000s）——Tier2/Tier3跟进

电装泰国需要连接器，于是日本端子（Nihon Tanshi）也去了。爱信需要铸件，于是旭铁工也去了。每家Tier1拉3-5家Tier2，每家Tier2可能再带1-2家Tier3。雪球越滚越大。

第四波（2000s-至今）——服务业跟进

制造业人多了，日系银行（三菱UFJ、三井住友、瑞穗）开分行，日系物流（日本通运、近铁Express）设仓库，日系人才公司（Recruit、Pasona）开办事处，甚至日系拉面店、居酒屋、便利店（7-11在泰国就是日系主导）都跟着来了。

最终结果： 丰田1家进入泰国 → 60年后泰国有6,083家日系企业。这不是6,083个独立决策，而是一张蛛网的自然延伸。

会在梦中，自动整理你让Claude存下的所有记忆，以避免记忆紊乱。

4. Daemon，变身后台服务

Daemon模式让Claude Code变成一个后台守护进程，就像数据库服务或者Web服务器那样，一直在后台跑着。

还是小龙虾的特性。

5. UDS Inbox，跨会话通信

这个功能可以让不同的Claude Code会话之间可以互相发消息。

比如你开了三个Claude Code窗口，以前它们是完全隔离的，互相不知道对方的存在，有了UDS Inbox之后，它们可以互相通信了。

6. Teleport，跨机器传送

Claude Code源代码泄露对我很友好，把整个工作会话从一台电脑传送到另一台，把所有上下文全部打包，可以公司电脑干到一半，回家打开Claude Code继续，不需要每天背着电脑跑。

你输入/ultraplan加一段描述，系统会在云端启动一个远程Claude Code实例，用Opus 4.6模型，花最多30分钟来探索你的项目、分析可行性、规划步骤，可以提前避坑。

8. Ultrareview

输入/ultrareview，系统在云端启动一支Bug舰队，默认5个Agent出动，最多20个，每个从不同角度审查你的代码，整个过程10到25分钟，纯粹的暴力美学。

第一层：往输出里掺假。API 返回结果时，服务端会混入一些虚假的工具调用数据。正常用户完全不受影响，服务端会帮你过滤掉。但如果有人批量抓取 API 输出去训练模型，这些假数据会一起被吃进训练集，污染模型质量。

第二层：把推理过程藏起来。Claude 在工具调用之间会产生中间推理文本，比如"我先读这个文件，再检查语法"。这些细节对蒸馏训练价值很高，因为它暴露了模型怎么思考。这一层把中间文本替换为一句摘要加密签名，下一轮对话时客户端拿签名换回原文。外部观察者只能看到摘要，完整推理链拿不到。

第三层：协议隔离。Claude Code 用一种新的 JSON 协议格式跟 API 通信，带独立的版本标记，跟每周 920 万次的普通 API 请求在统计上隔离开。服务端可以对不同群体做差异化处理，也让竞争对手无法简单伪装成 Claude Code 来获取特殊待遇。附带好处是省了大约 4.5% 的输出 token。
